Setup guide · macOS / Linux / Windows (WSL) · ~10 minutes

claudex = Claude Code harness × GPT-5.6 Sol

Run the Claude Code CLI — same tools, skills, and UI — with OpenAI's GPT-5.6 Sol as the model, billed against your existing ChatGPT/Codex subscription. Your normal claude command stays untouched; claudex is a separate escape hatch for when Anthropic usage runs out.

00 Prerequisites

  • Claude Code installed on any OS: curl -fsSL https://claude.ai/install.sh | bash (macOS/Linux/WSL), or brew install --cask claude-code (the plain claude cask is the desktop app, not the CLI), or npm install -g @anthropic-ai/claude-code.
  • An OpenAI account with Codex access (ChatGPT Plus/Pro). You do not need the Codex CLI installed — just the account.
  • A browser on the machine for a one-time OAuth login (headless servers: use --no-browser in step 04 and open the printed URL elsewhere).
  • On Windows, do everything inside WSL2 and follow the Linux variants.

01 Install CLIProxyAPI

CLIProxyAPI is a local proxy that speaks the Anthropic Messages API on one side and OpenAI's Codex backend on the other. Claude Code thinks it's talking to Anthropic; the proxy translates.

● macOS (Homebrew)

brew install cliproxyapi

● Linux / WSL (release binary)

# grab the latest release for your arch from
# https://github.com/router-for-me/CLIProxyAPI/releases
mkdir -p ~/.local/bin ~/.config/cliproxyapi
tar -xzf CLIProxyAPI_*_linux_*.tar.gz
mv cli-proxy-api ~/.local/bin/cliproxyapi
cp config.example.yaml ~/.config/cliproxyapi/config.yaml
# make sure ~/.local/bin is on your PATH

02 Configure the proxy

Generate an API key for the proxy (this is a local secret you invent, not an OpenAI key). Save the output — it's needed in steps 05 and 06:

openssl rand -hex 24

Open the config file in an editor:

PlatformConfig file
macOS (Homebrew)$(brew --prefix)/etc/cliproxyapi.conf — i.e. /opt/homebrew/etc/… on Apple Silicon, /usr/local/etc/… on Intel
Linux / WSL~/.config/cliproxyapi/config.yaml (passed via -config in step 03)

and change three things:

# 1. bind to localhost only (default is all interfaces)
host: "127.0.0.1"

# 2. replace the placeholder api-keys with your generated key
api-keys:
  - "<paste the openssl output here>"

# 3. add near the bottom — expose effort-tier variants of the model and
#    force a reasoning effort per tier
oauth-model-alias:
  codex:
    - name: "gpt-5.6-sol"
      alias: "gpt-5.6-sol-low"
      fork: true
    - name: "gpt-5.6-sol"
      alias: "gpt-5.6-sol-medium"
      fork: true
    - name: "gpt-5.6-sol"
      alias: "gpt-5.6-sol-xhigh"
      fork: true

payload:
  override:
    - models:
        - name: "gpt-5.6-sol"
          protocol: "codex"
      params:
        "reasoning.effort": "high"
    - models:
        - name: "gpt-5.6-sol-low"
          protocol: "codex"
      params:
        "reasoning.effort": "low"
    - models:
        - name: "gpt-5.6-sol-medium"
          protocol: "codex"
      params:
        "reasoning.effort": "medium"
    - models:
        - name: "gpt-5.6-sol-xhigh"
          protocol: "codex"
      params:
        "reasoning.effort": "xhigh"

This creates three client-visible names for the same upstream model, each pinned to a reasoning effort at the proxy — no matter what the client sends. The point: the main session (the orchestrator) runs gpt-5.6-sol at high, while agents and background calls remapped to -low/-medium (step 06) get cheaper effort automatically. Verified: the proxy's override wins even when the client requests a different effort.

03 Start it as a startup service

Run once; after this the proxy starts itself at every boot/login, forever. Set-and-forget — you never start it manually again (see step 08).

● macOS (Homebrew → launchd)

brew services start cliproxyapi
# after any config edit:
brew services restart cliproxyapi

● Linux / WSL (systemd user unit)

mkdir -p ~/.config/systemd/user
cat > ~/.config/systemd/user/cliproxyapi.service <<'EOF'
[Unit]
Description=CLIProxyAPI

[Service]
ExecStart=%h/.local/bin/cliproxyapi -config %h/.config/cliproxyapi/config.yaml
Restart=on-failure

[Install]
WantedBy=default.target
EOF
systemctl --user enable --now cliproxyapi
sudo loginctl enable-linger $USER   # keep it running from boot, not just login
# after any config edit:
systemctl --user restart cliproxyapi

04 Connect your OpenAI account

cliproxyapi --codex-login
# Linux: add -config ~/.config/cliproxyapi/config.yaml
# headless box: add --no-browser and open the printed URL on another device

A browser tab opens at auth.openai.com — sign in and authorize. Credentials land in ~/.cli-proxy-api/ (a codex-<email>.json file) and refresh themselves from then on. This is one-time.

05 Verify the proxy sees the models

curl -s http://127.0.0.1:8317/v1/models \
  -H "Authorization: Bearer <your-generated-key>" | python3 -m json.tool

You should see gpt-5.6-sol in the list, plus your gpt-5.6-sol-low and gpt-5.6-sol-medium forks from step 02 (and gpt-5.6-luna, gpt-5.5, and friends). An empty data: [] means step 04 didn't complete.

06 Add the alias

Append to your shell's rc file — ~/.zshrc for zsh (macOS default), ~/.bashrc for bash (most Linux/WSL defaults); echo $SHELL tells you which. Substitute your generated key from step 02:

# claudex — Claude Code harness driving GPT-5.6 Sol via CLIProxyAPI
alias claudex='ANTHROPIC_BASE_URL=http://127.0.0.1:8317 \
ANTHROPIC_AUTH_TOKEN=<your-generated-key> \
ANTHROPIC_DEFAULT_HAIKU_MODEL=gpt-5.6-sol-low \
ANTHROPIC_DEFAULT_SONNET_MODEL=gpt-5.6-sol-low \
ANTHROPIC_DEFAULT_OPUS_MODEL=gpt-5.6-sol-medium \
ANTHROPIC_DEFAULT_FABLE_MODEL=gpt-5.6-sol-xhigh \
CLAUDE_CODE_MAX_CONTEXT_TOKENS=250000 \
CLAUDE_CODE_ALWAYS_ENABLE_EFFORT=1 \
CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY=3 \
ENABLE_TOOL_SEARCH=true \
claude --model gpt-5.6-sol'

Then reload the shell (source ~/.zshrc / source ~/.bashrc, or just open a new terminal). What each variable does:

VariablePurpose
ANTHROPIC_BASE_URLPoints Claude Code at the local proxy instead of Anthropic.
ANTHROPIC_AUTH_TOKENAuthenticates to the proxy with your generated key.
ANTHROPIC_DEFAULT_HAIKU_MODEL
ANTHROPIC_DEFAULT_SONNET_MODEL
ANTHROPIC_DEFAULT_OPUS_MODEL
ANTHROPIC_DEFAULT_FABLE_MODEL		Remap every Claude model alias to an effort tier of GPT-5.6 Sol: haiku (background chores like session titles) and sonnet-pinned agents → -low; opus-pinned agents → -medium; fable-pinned agents (e.g. a Socratic gate) → -xhigh; the main session stays on the base name at high. Covers agent definitions that pin model: in frontmatter, so nothing in the session can accidentally try to reach Anthropic. Don't set CLAUDE_CODE_SUBAGENT_MODEL — it would force one model onto all subagents and defeat the tiers.
CLAUDE_CODE_MAX_CONTEXT_TOKENSDeclares the model's effective context budget (applies only to non-claude-* model IDs) — ~250k, matching Codex's live-session cap; see the context-window section below.
CLAUDE_CODE_ALWAYS_ENABLE_EFFORTEnables reasoning-effort controls for non-Anthropic models.
CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCYCaps parallel tool calls at 3 — the Codex backend handles bursts poorly.
ENABLE_TOOL_SEARCHForced on — defers MCP tool schemas behind a search index instead of eager-loading them (~30k tokens reclaimed with a typical MCP setup). Verified: GPT-5.6 Sol discovers and calls deferred tools via ToolSearch through the proxy. Explicit true matters: auto-defer only kicks in above ~10% of the context budget.

Because the variables are inline in the alias, they apply to that invocation only. Plain claude keeps using your Anthropic login. Extra arguments pass through: claudex --continue, claudex -p "…" all work.

07 Smoke test

claudex -p "Reply with exactly: claudex works."

✓ A reply means the whole chain works. You'll also see a warning that claude.ai connectors are disabled — expected, since the auth token overrides your claude.ai login for that session.

§ Reading the startup banner

gpt-5.6-sol with high effort · API Usage Billing

"API Usage Billing" does not mean you're paying per-token. The label only means Claude Code is authenticating with an ANTHROPIC_AUTH_TOKEN instead of your claude.ai login — from its point of view, that's "API mode." But the token is the local proxy's key, the "API" is 127.0.0.1:8317, and the proxy pays for nothing: it forwards to OpenAI's Codex backend using your ChatGPT OAuth, so usage comes out of your ChatGPT subscription quota.

  • No Anthropic API charges — requests never touch Anthropic.
  • No OpenAI API charges — it's not an API key, it's your subscription login.

The effort shown in the banner is Claude Code's client-side setting — change it in-session with /effort. Either way, the proxy's per-tier reasoning.effort overrides from step 02 win upstream: base model high, -medium medium, -low low, regardless of what the client sends.

08 Nothing to re-do after a restart

Every piece of this setup is persistent. After a reboot, opening a terminal and typing claudex just works — there is no daily ritual.

The only one-time actions are steps 01–07 themselves. If claudex ever fails after a reboot, the troubleshooting table below covers the two realistic causes: the service not running, or a stale shell.

§ Context window: declare the real size, never [1m]

The Codex models' window is 400k tokens total, split ~272k input + 128k output/reasoning — and in live coding sessions the effective usable context caps around ~258k. Raw single-shot probes accept more input (a 328k-input request with tiny output passes, because only the 400k total is checked), which is misleading: a real session that grows past ~258k starts hitting context errors. Claude Code doesn't know any of this — it budgets non-Anthropic models at 200k by default, and its [1m] model-ID suffix budgets 1M, which lets long sessions crash far before auto-compact would ever trigger.

The fix is CLAUDE_CODE_MAX_CONTEXT_TOKENS=250000 in the alias — a Claude Code override that sets the context budget for any model whose ID doesn't start with claude-. 250k sits just under the ~258k effective cap, so auto-compact fires while sessions are still safely inside the window; /context confirms the budget (30.9k / 250k), and auto-compact + /compact are verified working through the proxy. (We ran 340k for a while on the strength of the raw probes — real sessions occasionally errored near the top; 250k matches how Codex itself caps live sessions.)

Don't "upgrade" this to [1m] or past ~258k — bigger numbers route fine and look like they work, right up until a long run crosses the effective cap and dies mid-task. Custom suffixes like [400k] aren't supported (silent fallback to 200k); the env var is the supported mechanism.
