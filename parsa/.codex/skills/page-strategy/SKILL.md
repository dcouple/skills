# Page Strategy

Use this skill before drafting a page. The job is to decide what page deserves to exist, what it must accomplish for the searcher, and how the product can appear without distorting the answer.

Core rule: promotion is allowed only when it helps the searcher's job-to-be-done. If the page would collapse after removing CTAs, the strategy is weak.

## Inputs To Gather

- Target keyword/query or topic cluster.
- Business goal and desired conversion, if any.
- Product, category, or offer being connected to the query.
- Audience and buying stage.
- Current SERP notes, competitor examples, or pages the user admires/dislikes.
- Constraints: brand voice, claims allowed, proof available, screenshots/data/examples available.

If key inputs are missing, make reasonable assumptions and label them. Ask only when the answer changes the page type, product role, or risk profile.

## Workflow

1. Classify the searcher job and dominant/secondary intent. Load `references/intent-and-searcher-job.md`.
2. Decide whether the page should exist for this brand. Load `references/product-fit-permission.md`.
3. Pick the page type, answer shape, and expected comparison set.
4. Set the time-to-value bar. Load `references/time-to-value.md`.
5. Define proof/E-E-A-T requirements: firsthand use, screenshots, data, expert input, citations, examples, pricing, limitations, or methodology.
6. Define product role and CTA boundaries: lead, contextual, related, light mention, or exclude.
7. Draft the page brief. Use `references/page-strategy-workflow.md` for the output structure.
8. When the page is a listicle, comparison, alternatives page, or product-led informational page, load `references/good-vs-bad-examples.md` and calibrate against the patterns.

## Output Format

Return a practical page brief:

- Page thesis
- Target query and intent map
- Audience/job-to-be-done
- Recommended page type
- Time-to-value requirement
- Product-fit permission and CTA boundary
- Required proof/E-E-A-T
- Recommended structure
- SERP differentiation angle
- What not to include
- Risks before writing
- Handoff notes for `$page-review`

## Handoffs

- Use `$page-review` after a draft exists.
- Use `$site-content-audit` when the question involves many URLs, a sitemap, a traffic decline, a competitor teardown, or pruning/merging decisions.
